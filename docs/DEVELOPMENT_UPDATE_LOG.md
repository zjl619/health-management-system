# 开发改进记录

## 文档信息

- 项目：健康管理小程序
- 记录目的：沉淀本轮从前端单机版到前后端联通版的改造过程
- 时间范围：2026-04

## 1. 改进背景

项目早期以本地缓存为主，具备页面功能但缺少后端承载能力，存在以下问题：

- 数据无法跨设备同步
- 用户身份不可追踪
- 统计口径分散在前端各页面
- 环境搭建依赖 SQL Server，启动门槛较高

## 2. 本轮改进目标

1. 建立可运行的后端接口体系。
2. 接通前端 services/api 到后端接口。
3. 增加用户登录与资料维护能力。
4. 落地统计接口，减少前端重复计算。
5. 将数据库方案切换为本地 SQLite，降低环境复杂度。

## 3. 核心改动清单

### 3.1 后端能力新增

- 新增 auth 路由：登录并签发 token
- 新增 users 路由：查询与更新个人资料
- 新增 stats 路由：summary、trends、mood-distribution
- 增加鉴权中间件与 token 工具

### 3.2 前端能力新增

- 新增 authApi、statsApi
- 新增 authService、statsService
- 统计页改为读取后端统计接口
- 新增个人资料页，支持查看与修改资料

### 3.3 数据层改造

- 后端数据库由 SQL Server 改为 SQLite（sql.js）
- 启动时自动初始化 users、health_records、diet_entries、foods 表
- SQLite 文件落地到 server/data/health-app.sqlite

## 4. 文档与运维改进

- README 与现状统一，明确 SQLite 启动流程
- 架构文档从规划稿更新为现状说明
- SQL 参考脚本改为 SQLite 语法

## 5. 验证结果

本轮改进完成后，已验证以下链路可用：

1. 后端健康检查接口可响应。
2. 登录接口可返回 token。
3. 鉴权后可读取和更新用户资料。
4. 统计接口可返回汇总数据。
5. 小程序可完成核心页面访问与数据交互。

## 6. 已知限制

- 登录方案为本地 openid，尚未接微信真实登录。
- 食物库后端管理能力仍待补齐。
- 同步冲突处理、离线队列、自动化测试未完善。

## 7. 下一步计划

1. 完成 foods 的增删改接口与前端管理入口。
2. 增加核心接口自动化测试（auth、health、diet、stats）。
3. 统一错误码和日志格式。
4. 补充部署、备份与恢复指南。

## 8. 经验总结

- 先打通端到端链路，再做架构深化，效率更高。
- 文档应和代码同节奏更新，否则会影响联调和验收。
- 对课程项目来说，降低环境门槛比引入复杂基础设施更重要。
